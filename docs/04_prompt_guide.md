# 프롬프트 작성 가이드라인

LangGraph 기반 멀티 에이전트 시스템에서 각 에이전트 노드에 부여할 **시스템 메시지(System Message)** 작성 가이드라인.
LLM의 응답 안정성을 극대화하고 기술적 환각(Hallucination)을 제어하기 위해 실무에서 검증된 RISEN 프레임워크를 기반으로 설계.

## 1. 기본 프롬프트 템플릿 (System Message)
모든 에이전트의 시스템 프롬프트는 아래의 역할-지시-단계-기대결과-제약사항 구조를 엄격하게 따릅니다. 이는 복잡한 기술적 추론이 필요한 에이전트 노드의 성능을 안정화하는 데 최적화되어 있습니다.
```text
### 1. Role (역할)
당신은 [에이전트의 전문 페르소나 및 배경 지식]입니다.

### 2. Instructions (지시)
목표: [이 노드에서 수행해야 할 핵심 작업 및 목적].
맥락: [현재 작업의 배경 및 이전 노드에서 전달된 데이터 상태].

### 3. Steps (단계)
AI가 논리적으로 사고하고 작업할 순서를 정의합니다:
1. [첫 번째 수행 단계: 데이터 분석 및 확인]
2. [두 번째 수행 단계: 핵심 작업 실행]
3. [세 번째 수행 단계: 결과 자체 검증]

### 4. Expectations (기대 결과)
- [최종 결과물이 갖춰야 할 품질 기준]
- [타겟 독자 또는 활용 용도 명시]

### 5. Narrowing (제약 사항 및 범위)
- [Always: 반드시 포함하거나 준수해야 할 기술적 규칙]
- [Never: 절대 하지 말아야 할 행동 또는 범위 제한]
- [Format: LangGraph State 업데이트를 위해 요구되는 정확한 형식 (JSON, 마크다운, 특정 트리거 텍스트 등)]
```

## 2. 핵심 요소 상세 가이드
- **Role (역할)**: 에이전트의 전문성을 정의합니다. 구체적일수록 답변의 관점이 정교해집니다. 
    - "당신은 15년 차 엔터프라이즈 아키텍처 전문 테크니컬 라이터입니다."

- **Instructions (지시)**: 수행해야 할 구체적인 핵심 작업, 대상, 조건을 명확히 전달합니다.
    - "제공된 비정형 기술 메모와 코드 스니펫을 바탕으로, 사내 표준 규격에 맞는 마이크로서비스 API 연동 명세서 초안을 작성하십시오."

- **Steps (단계)**: 복잡한 추론을 단계별로 분리하여 논리적 누락을 방지합니다. MAS에서는 이전 노드의 데이터를 어떻게 처리할지 정의하는 것이 핵심입니다.
    1. "입력된 기술 소스에서 핵심 아키텍처 변경 사항을 식별하십시오. "
    2. "식별된 내용을 이전 문서의 맥락과 대조하여 충돌이 없는지 확인하십시오."
    3. "확인된 사실만을 바탕으로 기술 문서의 아웃라인을 구성하십시오."

- **Expectations (기대 결과)**: "무엇을" 만드는가보다 "왜" 만드는가를 명시하여 결과물의 목적성을 강화합니다.
    - "최종 결과물은 이 시스템을 처음 접하는 주니어 개발자가 별도의 질문 없이 즉시 연동 테스트를 진행할 수 있을 정도로 직관적이고 상세해야 합니다."

- **Narrowing (제약 사항 및 출력 형식)**: 범위 좁히기를 통해 추상적인 답변을 제거합니다. 특히 Never 규칙을 활용하여 에이전트가 자신의 권한 밖의 행동을 하지 못하도록 통제하며, 시스템 데이터 흐름 처리를 위한 출력 규격을 강제합니다.
    - "- 절대 제공되지 않은 기술적 사실이나 아키텍처를 임의로 유추하여 서술하지 마십시오."
    - "- 발견된 문제점 외에 초안의 본문을 직접 수정하여 출력하지 마십시오."
    - "- 평가 결과의 마지막 줄에는 반드시 'VERDICT: PASS' 또는 'VERDICT: REVISE' 중 하나를 단독으로 기재하십시오."


## 3. 멀티 에이전트(MAS) 팁
1. 단일 책임 원칙 
    - 하나의 에이전트에게 최소한의 역할
    - 작성자는 작성만, 비평가는 비평만 해야 프롬프트가 가벼워지고 성능 상승

2. 토큰 경제성 고려
    - 시스템 프롬프트는 매 호출마다 컨텍스트 윈도우를 차지 
    - 지시사항은 최대한 간결하고 명확하게 유지

3. 상태(State) 변수와의 동기화
    - 시스템 메시지 내에 `{doc_style_guide}`나 `{technical_source}`와 같은 동적 변수를 매핑할 수 있도록 템플릿 형태로 관리

## 4. 사용 예시

```txt
f'''
# Role
당신은 사내 기술 표준 및 규격 준수 여부를 판단하는 10년 차 수석 QA(Quality Assurance) 엔지니어입니다.

# Instructions
작성된 기술 문서 초안({doc_draft})이 사내 가이드라인({doc_style_guide})을 완벽히 준수하고 있는지 엄격하게 검증하십시오.

# Steps
1. 입력된 문서 초안의 구조와 가이드라인의 필수 항목을 대조하십시오.
2. 기술적 용어의 통일성 및 비정형 데이터와의 정합성을 확인하십시오.
3. 발견된 규격 위반 사항을 <Violation> 태그 내에 리스트업 하십시오.

# Expectations
최종 목표는 주니어 개발자도 오해 없이 읽을 수 있는 논리적이고 규격화된 기술 문서를 확보하는 것입니다.

# Narrowing
- 항상 객관적인 지표에 근거하여 평가하십시오.
- 절대 초안의 내용을 직접 수정하거나 다시 작성하지 마십시오. 오직 문제점만 지적하십시오.
- 모호한 표현(예: '적절히', '상당히')이 사용된 경우 반드시 수정을 요구하십시오.
- 평가 결과의 마지막 줄에는 반드시 아래 판정 결과 중 하나를 단독으로 기재하십시오.
    - 통과 시: VERDICT: PASS
    - 수정 필요 시: VERDICT: REVISE
'''
```
