# 프롬프트 작성 가이드라인

LangGraph 기반 멀티 에이전트 시스템에서 각 에이전트 노드에 부여할 **시스템 메시지(System Message)** 작성 가이드라인.
LLM의 응답 안정성을 높이고 할루시네이션을 제어하기 위해 실무에서 검증된 CARE/RISEN 프레임워크를 기반으로 작성.

## 1. 기본 프롬프트 템플릿 (System Message)
모든 에이전트의 시스템 프롬프트는 아래의 **역할-목표-맥락-규칙-출력 형식** 구조를 엄격하게 따릅니다.
```text
당신은 [Role: 역할 및 페르소나]입니다.
목표: [Objective: 이 노드에서 달성해야 할 단일 목표].
맥락: [Context: 현재 작업의 배경이나 이전 노드에서 넘어온 데이터 설명].

행동 지침 (Rules):
- [규칙 1: Always - 반드시 지켜야 할 사항]
- [규칙 2: Never - 절대 하지 말아야 할 행동 (경계 설정)]
- [규칙 3: Tone & Manner 등 스타일 제약]

예시 (Few-Shot): 
- [Example: 기대하는 입력과 출력의 패턴 (필요시 포함)]

출력 형식 (Format):
- [Format: LangGraph State 업데이트를 위해 요구되는 정확한 형식 (예: 특정 마크다운, 특정 단어 포함 등)]
```

## 2. 핵심 요소 상세 가이드
- **Role (역할)**: 에이전트의 전문성을 정의하여 답변의 품질 상승 (예: "당신은 10년 차 IT 전문 블로그 편집장입니다.")

- **Rules (행동 지침)**: 
    - "항상(always)/절대(never)"와 같은 강한 어휘를 사용하여 에이전트의 권한을 통제
    - 멀티 에이전트 시스템에서는 자신의 역할 범위를 넘어서는 행동을 막는 것이 핵심

- **나쁜 예**: "글을 검토하고 고쳐주세요."

- **좋은 예**: "초안을 평가만 하세요. 절대 초안을 직접 수정하거나 재작성하지 마세요."

- **Format (출력 형식)**: 시스템 라우팅에 필요한 트리거 단어를 명시 (예: "통과 시 반드시 문서 끝에 'VERDICT: OK'를 작성하세요.")

- **Example (예시)**: LLM이 지시를 오해할 여지가 있는 복잡한 작업(예: JSON 파싱 구조 등)인 경우 1~2개의 샘플을 제공하여 패턴을 학습 (Few-shot 엔지니어링)

## 3. 멀티 에이전트(MAS) 팁
1. 단일 책임 원칙 
    - 하나의 에이전트에게 최소한의 역할
    - 작성자는 작성만, 비평가는 비평만 해야 프롬프트가 가벼워지고 성능 상승

2. 토큰 경제성 고려
    - 시스템 프롬프트는 매 호출마다 컨텍스트 윈도우를 차지 
    - 지시사항은 최대한 간결하고 명확하게 유지

3. 상태(State) 변수와의 동기화
    - 시스템 메시지 내에 {tone_and_manner}나 {accumulated_context}와 같은 동적 변수를 매핑할 수 있도록 템플릿 형태로 관리

## 4. 사용 예시

```txt
f'''
**Role:**
당신은 10년 차 IT 테크 블로그 전문 편집장입니다.

**Objective:**
작성된 블로그 포스트 초안이 주어진 톤앤매너를 유지하고 있는지, 논리적 비약이나 기술적 오류는 없는지 엄격하게 평가하세요.

**Context:**
- 타겟 독자: AI 기술에 관심 있는 주니어 개발자 및 기획자
- 요구되는 톤앤매너: {tone_and_manner}

**Rules:**
- 항상 초안의 문맥 흐름과 일관성에 집중해서 평가하세요.
- 절대 초안의 내용을 직접 수정하거나 전체를 다시 작성하지 마세요. (오직 문제점만 지적할 것)
- 피드백은 3문장 이내로 핵심만 간결하게 작성하세요.

**Output format:**
- 피드백 내용의 가장 마지막 줄에는 반드시 아래 둘 중 하나의 판정 결과를 정확한 영문 텍스트로 기재하세요. (줄바꿈 후 단독으로 작성)
  - 통과 시: VERDICT: OK
  - 수정 필요 시: VERDICT: REVISE
'''
```
